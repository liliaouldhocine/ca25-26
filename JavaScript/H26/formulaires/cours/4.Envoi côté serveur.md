# Support de cours

# Envoi avec `fetch` et réception côté serveur avec Express

## Présentation générale

Dans les cours précédents, nous avons appris à :

```txt
1. Construire un formulaire HTML.
2. Soumettre un formulaire.
3. Empêcher le comportement par défaut avec event.preventDefault().
4. Récupérer les données avec JavaScript.
5. Valider les données côté client.
6. Afficher des messages d’erreur.
```

Nous allons maintenant passer à une étape très importante : **envoyer les données du formulaire à un serveur**.

Dans une vraie application web, les données ne restent pas seulement dans le navigateur. Elles peuvent être envoyées à une API, puis reçues par un serveur, traitées, validées, enregistrées ou utilisées pour produire une réponse.

Dans ce cours, nous allons étudier deux grandes parties :

```txt
1. Côté navigateur : envoyer les données avec fetch.
2. Côté serveur : recevoir les données avec Express.
```

`fetch` est une API JavaScript moderne qui permet de faire des requêtes HTTP depuis le navigateur. Elle est basée sur les promesses et permet de récupérer ou d’envoyer des ressources à travers le réseau. ([MDN Web Docs][1])

Express est un framework Node.js minimaliste et flexible qui permet de créer des applications web et des API. Il fournit notamment un système de routes et de middlewares pour recevoir et traiter des requêtes HTTP. ([expressjs.com][2])

---

# 1. Pourquoi utiliser `fetch` ?

Jusqu’ici, lorsqu’un formulaire était soumis, nous pouvions récupérer les données et les afficher dans la console.

Exemple :

```javascript
const formData = new FormData(form);
const donnees = Object.fromEntries(formData.entries());

console.log(donnees);
```

Mais dans une vraie application, afficher les données dans la console ne suffit pas.

On veut souvent envoyer les données à un serveur pour :

- créer un compte ;
- enregistrer une inscription ;
- envoyer un message de contact ;
- ajouter un commentaire ;
- modifier un profil ;
- enregistrer une commande ;
- sauvegarder des préférences ;
- communiquer avec une base de données.

`fetch` permet justement d’envoyer une requête depuis le navigateur vers une adresse précise.

---

# 2. Comprendre le dialogue navigateur-serveur

Quand un utilisateur remplit un formulaire, il se passe souvent ceci :

```txt
Utilisateur
   ↓
Formulaire HTML
   ↓
JavaScript récupère les données
   ↓
fetch envoie les données
   ↓
Serveur Express reçoit la requête
   ↓
Express traite les données
   ↓
Express renvoie une réponse
   ↓
JavaScript affiche le résultat dans la page
```

Ce dialogue est au cœur des applications web modernes.

Le navigateur ne travaille donc pas seul.
Le serveur ne travaille pas seul non plus.
Les deux communiquent grâce aux requêtes HTTP.

---

# 3. Rappel : qu’est-ce qu’une requête HTTP ?

Une requête HTTP est un message envoyé par le navigateur vers un serveur.

Elle contient généralement :

```txt
1. Une URL
2. Une méthode HTTP
3. Des headers
4. Parfois un body
```

Exemple simplifié :

```txt
POST /api/contact
Content-Type: application/json

{
  "nom": "Sara",
  "email": "sara@email.com",
  "message": "Bonjour"
}
```

Dans cet exemple :

```txt
POST                → méthode HTTP
/api/contact        → destination
application/json    → format des données
body                → données envoyées
```

---

# 4. GET et POST : rappel rapide

Nous avons déjà vu que les méthodes GET et POST n’ont pas le même rôle.

## GET

GET sert surtout à demander de l’information.

Exemples :

- chercher des articles ;
- filtrer une liste ;
- consulter des résultats ;
- récupérer des données.

Exemple :

```javascript
fetch("/api/articles");
```

Ici, on demande des articles au serveur.

---

## POST

POST sert surtout à envoyer des données.

Exemples :

- envoyer un formulaire ;
- créer une inscription ;
- ajouter un commentaire ;
- envoyer un message ;
- créer une ressource.

Exemple :

```javascript
fetch("/api/contact", {
  method: "POST",
  body: ...
});
```

Ici, on envoie des données au serveur.

---

# 5. Première requête avec `fetch`

La forme la plus simple de `fetch` ressemble à ceci :

```javascript
fetch("/api/message");
```

Cette ligne envoie une requête vers :

```txt
/api/message
```

Par défaut, si aucune méthode n’est précisée, la requête est généralement une requête GET.

Cependant, `fetch` retourne une promesse. Cela signifie que la réponse n’arrive pas instantanément. Le navigateur doit attendre que le serveur réponde.

---

# 6. Utiliser `fetch` avec `async` et `await`

Pour écrire un code plus lisible, on utilise souvent `async` et `await`.

```javascript
async function chargerMessage() {
  const response = await fetch("/api/message");
  const data = await response.json();

  console.log(data);
}
```

Explication :

```javascript
const response = await fetch("/api/message");
```

Cette ligne attend la réponse du serveur.

```javascript
const data = await response.json();
```

Cette ligne lit le corps de la réponse et le transforme en objet JavaScript.

La méthode `fetch()` retourne une promesse qui se résout avec un objet `Response`. Ensuite, on utilise des méthodes comme `response.json()` pour lire le contenu de la réponse selon le format attendu. ([MDN Web Docs][3])

# 9. Préparer les données en JSON

  email: "sara@email.com",

Mais pour l’envoyer dans une requête HTTP, on le transforme en texte JSON :

const donneesJSON = JSON.stringify(donnees);

  "message": "Bonjour"

```txt

```

# 10. Structure d’un envoi POST avec `fetch`

Voici la structure générale :

```javascript
const response = await fetch("/api/contact", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(donnees),
});
```

Lecture du code :

```txt
/api/contact
```

C’est l’adresse du serveur qui recevra les données.

```javascript
method: "POST";
```

On indique que l’on envoie des données.

```javascript
headers: {
  "Content-Type": "application/json"
}
```

On indique que les données envoyées sont au format JSON.

```javascript
body: JSON.stringify(donnees);
```

On place les données dans le corps de la requête.

---

# 11. Exemple complet : formulaire envoyé avec `fetch`

```javascript
const form = document.querySelector("#contactForm");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    resultat.textContent = data.message;
  } catch (error) {
    resultat.textContent = "Une erreur est survenue pendant l’envoi.";
    console.error(error);
  }
});
```

Ce code fait plusieurs choses :

```txt
1. Il écoute la soumission du formulaire.
2. Il empêche le rechargement de la page.
3. Il récupère les données avec FormData.
4. Il transforme les données en objet.
5. Il envoie les données avec fetch.
6. Il attend la réponse du serveur.
7. Il affiche le message du serveur dans la page.
8. Il gère les erreurs avec try/catch.
```

---

# 12. Pourquoi utiliser `try...catch` ?

Une requête peut échouer pour plusieurs raisons :

- le serveur n’est pas lancé ;
- l’adresse est incorrecte ;
- la connexion est interrompue ;
- le serveur répond avec une erreur ;
- la réponse n’est pas au format attendu.

`try...catch` permet d’éviter que l’application s’arrête brutalement.

Exemple :

```javascript
try {
  const response = await fetch("/api/contact");
  const data = await response.json();

  console.log(data);
} catch (error) {
  console.error("Erreur :", error);
}
```

Le `catch` récupère l’erreur si quelque chose se passe mal pendant la requête.

---

# 13. Vérifier `response.ok`

Une subtilité importante : `fetch` ne considère pas automatiquement toutes les réponses HTTP d’erreur comme des erreurs JavaScript.

Par exemple, si le serveur répond avec un statut `400` ou `500`, la requête peut quand même produire une réponse. Il est donc utile de vérifier :

```javascript
response.ok;
```

Exemple :

```javascript
const response = await fetch("/api/contact", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(donnees),
});

if (!response.ok) {
  throw new Error("La requête a échoué.");
}

const data = await response.json();
```

`response.ok` indique si le statut HTTP de la réponse correspond à une réussite.

# 14. Exemple plus robuste avec `response.ok`

```javascript
form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    if (!response.ok) {
      resultat.textContent =
        data.message || "Le formulaire contient une erreur.";
      return;
    }

    resultat.textContent = data.message;
    form.reset();
  } catch (error) {
    resultat.textContent = "Impossible de contacter le serveur.";
    console.error(error);
  }
});
```

Cette version est plus complète parce qu’elle distingue :

```txt
Erreur de validation côté serveur
Erreur de communication avec le serveur
Succès
```

# 15. Réception côté serveur avec Express

Maintenant que le navigateur envoie les données, il faut créer un serveur capable de les recevoir.

Express permet de définir des routes. Une route est une combinaison entre une méthode HTTP et un chemin. Par exemple, une route peut répondre à une requête GET sur `/about` ou à une requête POST sur `/api/contact`. ([expressjs.com][4])

Exemple :

```javascript
app.post("/api/contact", function (req, res) {
  // traitement
});
```

Cette route signifie :

```txt
Quand le serveur reçoit une requête POST vers /api/contact,
exécuter cette fonction.
```

# 22. Recevoir du JSON avec Express

Quand le navigateur envoie du JSON avec :

```javascript
headers: {
  "Content-Type": "application/json"
}
```

le serveur doit savoir lire ce JSON.

Dans Express, on ajoute le middleware :

```javascript
app.use(express.json());
```

Le middleware `express.json()` analyse les requêtes dont le `Content-Type` correspond à du JSON et place les données analysées dans `req.body`. ([expressjs.com][5])

# 24. Comprendre `req` et `res`

Dans Express, une route reçoit souvent deux paramètres :

```javascript
app.post("/api/contact", function (req, res) {
  // ...
});
```

## `req`

`req` représente la requête envoyée par le client.

Elle peut contenir :

```txt
req.body      → données envoyées dans le corps de la requête
req.query     → paramètres GET dans l’URL
req.params    → paramètres dynamiques dans la route
```

## `res`

`res` représente la réponse que le serveur renvoie au client.

On peut répondre avec :

```javascript
res.send("Texte simple");
```

ou :

```javascript
res.json({ message: "Réponse JSON" });
```

---

# 25. Pourquoi `req.body` peut être vide ?

Une erreur fréquente est d’oublier :

```javascript
app.use(express.json());
```

Si le navigateur envoie du JSON mais que le serveur ne contient pas ce middleware, Express ne sait pas encore comment lire le corps de la requête.

Résultat possible :

```javascript
req.body;
```

peut être vide ou `undefined`.

Il faut donc retenir :

```txt
fetch envoie du JSON
   ↓
Express doit utiliser express.json()
   ↓
Les données sont disponibles dans req.body
```

# 26. Validation côté serveur

Même si le formulaire a déjà été validé côté navigateur, il faut vérifier les données côté serveur.

Pourquoi ?

Parce que les données envoyées par un client ne sont jamais entièrement fiables.

Un utilisateur peut :

- désactiver JavaScript ;
- modifier le HTML ;
- envoyer une requête manuellement ;
- contourner les validations côté client.

La documentation Express rappelle aussi que `req.body` provient d’une entrée contrôlée par l’utilisateur et doit être considéré comme non fiable tant que le serveur ne l’a pas validé. ([expressjs.com][5])

---

# 27. Exemple de validation côté serveur

```javascript
app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();
  const email = req.body.email?.trim();
  const message = req.body.message?.trim();

  if (!nom || !email || !message) {
    return res.status(400).json({
      message: "Tous les champs sont obligatoires.",
    });
  }

  res.status(200).json({
    message: "Votre message a bien été reçu.",
  });
});
```

Explication :

```javascript
req.body.nom?.trim();
```

On récupère `nom`, puis on enlève les espaces inutiles.

Le `?.` évite une erreur si `nom` est absent.

```javascript
if (!nom || !email || !message)
```

On vérifie si une donnée est manquante.

```javascript
res.status(400);
```

On indique que la requête contient une erreur côté client.

```javascript
res.json(...)
```

On renvoie une réponse au format JSON.

# 28. Réponse de succès et réponse d’erreur

Un serveur doit répondre clairement.

## Réponse de succès

```javascript
res.status(200).json({
  message: "Votre message a bien été reçu.",
});
```

## Réponse d’erreur

```javascript
res.status(400).json({
  message: "Tous les champs sont obligatoires.",
});
```

Le navigateur peut ensuite lire cette réponse :

```javascript
const data = await response.json();

if (!response.ok) {
  resultat.textContent = data.message;
  return;
}

resultat.textContent = data.message;
```

# 31. Lecture complète du fonctionnement

Quand l’utilisateur clique sur “Envoyer” :

```txt
1. Le formulaire déclenche submit.
2. JavaScript bloque le rechargement avec preventDefault().
3. JavaScript récupère les données avec FormData.
4. Les données sont transformées en objet JavaScript.
5. L’objet est transformé en JSON.
6. fetch envoie une requête POST vers /api/contact.
7. Express reçoit la requête.
8. express.json() lit le JSON.
9. Les données deviennent disponibles dans req.body.
10. Le serveur valide les données.
11. Le serveur répond avec res.json().
12. Le navigateur lit la réponse avec response.json().
13. Le message est affiché dans la page.
```

# 32. Recevoir un formulaire HTML classique avec Express

Il existe une autre situation.

Si un formulaire HTML est soumis directement avec :

```html
<form action="/contact" method="POST"></form>
```

et sans `fetch`, les données ne sont pas envoyées en JSON. Elles sont généralement envoyées au format `application/x-www-form-urlencoded`.

Dans ce cas, côté Express, on utilise :

```javascript
app.use(express.urlencoded({ extended: true }));
```

Le middleware `express.urlencoded()` analyse les requêtes encodées en URL et place le résultat dans `req.body`. ([expressjs.com][5])

---

## Exemple

### HTML

```html
<form action="/contact-classique" method="POST">
  <label for="nom">Nom</label>
  <input type="text" id="nom" name="nom" />

  <label for="message">Message</label>
  <textarea id="message" name="message"></textarea>

  <button type="submit">Envoyer</button>
</form>
```

### Express

```javascript
app.use(express.urlencoded({ extended: true }));

app.post("/contact-classique", function (req, res) {
  console.log(req.body);

  res.send("Formulaire classique reçu.");
});
```

# 33. Différence entre `express.json()` et `express.urlencoded()`

| Situation                              | Middleware Express                       |
| -------------------------------------- | ---------------------------------------- |
| Données JSON envoyées avec `fetch`     | `express.json()`                         |
| Formulaire HTML classique sans `fetch` | `express.urlencoded({ extended: true })` |

## Exemple avec `fetch`

```javascript
fetch("/api/contact", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(donnees),
});
```

Côté serveur :

```javascript
app.use(express.json());
```

## Exemple avec formulaire HTML classique

```html
<form action="/contact" method="POST"></form>
```

Côté serveur :

```javascript
app.use(express.urlencoded({ extended: true }));
```

# 34. Cas des fichiers : attention

L’envoi de fichiers est différent.

Un formulaire avec fichier utilise souvent :

```html
enctype="multipart/form-data"
```

Exemple :

```html
<form action="/upload" method="POST" enctype="multipart/form-data">
  <input type="file" name="document" />
  <button type="submit">Envoyer</button>
</form>
```

Pour recevoir des fichiers avec Express, on utilise souvent un middleware spécialisé comme Multer. La documentation Express indique que Multer ajoute notamment un objet `body` et un objet `file` ou `files` à la requête, et rappelle d’utiliser `enctype="multipart/form-data"` dans le formulaire. ([expressjs.com][6])

Ce sujet peut être traité dans un cours séparé, car l’envoi de fichiers demande une logique différente.

# 35. Problème fréquent : `Cannot POST /api/contact`

Cette erreur signifie généralement que le serveur n’a pas de route POST correspondant à l’adresse demandée.

Exemple côté client :

```javascript
fetch("/api/contact", {
  method: "POST",
  ...
});
```

Il faut avoir côté serveur :

```javascript
app.post("/api/contact", function (req, res) {
  // ...
});
```

Si la route côté serveur est différente, par exemple :

```javascript
app.post("/contact", ...)
```

alors `/api/contact` ne correspond pas.

À vérifier :

```txt
L’URL dans fetch est-elle exactement la même que la route Express ?
La méthode est-elle bien POST des deux côtés ?
Le serveur est-il lancé ?
```

---

# 36. Problème fréquent : `req.body` est vide

Causes possibles :

## Cause 1 : middleware manquant

Si le client envoie du JSON :

```javascript
headers: {
  "Content-Type": "application/json"
}
```

il faut :

```javascript
app.use(express.json());
```

## Cause 2 : mauvais format envoyé

Si le client envoie du JSON, mais que le serveur attend du `urlencoded`, les données ne seront pas lues comme prévu.

## Cause 3 : champ sans `name`

Si on utilise `FormData`, les champs doivent avoir un attribut `name`.

Erreur :

```html
<input id="email" />
```

Correction :

```html
<input id="email" name="email" />
```

---

# 37. Problème fréquent : `Unexpected token < in JSON`

Cette erreur arrive souvent lorsque JavaScript essaie de lire une réponse avec :

```javascript
await response.json();
```

mais que le serveur ne répond pas avec du JSON.

Par exemple, le serveur renvoie une page HTML d’erreur au lieu de :

```json
{
  "message": "..."
}
```

À vérifier :

```txt
La route existe-t-elle ?
Le serveur répond-il avec res.json() ?
L’URL dans fetch est-elle correcte ?
La réponse est-elle vraiment du JSON ?
```

---

# 38. Problème fréquent : CORS

Si le client et le serveur ne sont pas sur la même origine, le navigateur peut bloquer la requête pour des raisons liées à CORS.

Exemple :

```txt
Client : http://localhost:5173
Serveur : http://localhost:3000
```

Dans ce cas, il peut être nécessaire de configurer CORS côté Express.

Exemple simple :

```bash
npm install cors
```

```javascript
import cors from "cors";

app.use(cors());
```

Le middleware CORS pour Express permet de configurer les autorisations entre origines différentes, et la documentation indique qu’il peut être appliqué au niveau de l’application avec `app.use(cors())`. ([expressjs.com][7])

Dans un premier projet pédagogique, on peut éviter ce problème en servant le fichier HTML directement avec Express grâce à :

```javascript
app.use(express.static("public"));
```

Ainsi, le client et le serveur utilisent la même origine :

```txt
http://localhost:3000
```

# 39. Mini-projet complet : formulaire de contact avec `fetch` et Express

## Structure du projet

```txt
contact-app/
│
├── server.js
├── package.json
│
└── public/
    ├── index.html
    └── script.js
```

---

## `package.json`

```json
{
  "name": "contact-app",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "start": "node server.js"
  },
  "dependencies": {
    "express": "^5.0.0"
  }
}
```

---

## `server.js`

```javascript
import express from "express";

const app = express();
const PORT = 3000;

app.use(express.static("public"));
app.use(express.json());

app.get("/api/status", function (req, res) {
  res.json({
    message: "API fonctionnelle.",
  });
});

app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();
  const email = req.body.email?.trim();
  const message = req.body.message?.trim();

  if (!nom || !email || !message) {
    return res.status(400).json({
      message: "Veuillez remplir tous les champs.",
    });
  }

  console.log("Message reçu :");
  console.log({
    nom: nom,
    email: email,
    message: message,
  });

  res.status(200).json({
    message: "Votre message a été envoyé avec succès.",
  });
});

app.listen(PORT, function () {
  console.log(`Serveur lancé sur http://localhost:${PORT}`);
});
```

---

<!DOCTYPE html>
<html lang="fr">

---

## `public/script.js`

```javascript
const form = document.querySelector("#contactForm");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  resultat.textContent = "";

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    if (!response.ok) {
      resultat.textContent = data.message;
      return;
    }

    resultat.textContent = data.message;
    form.reset();
  } catch (error) {
    resultat.textContent = "Impossible de contacter le serveur.";
    console.error(error);
  }
});
```

# 43. Bonnes pratiques importantes

## Toujours valider côté serveur

Même si le navigateur valide déjà le formulaire, le serveur doit vérifier les données.

```javascript
if (!nom || !email || !message) {
  return res.status(400).json({
    message: "Veuillez remplir tous les champs.",
  });
}
```

---

## Ne pas tout afficher dans les logs en production

Pendant l’apprentissage, il est utile d’écrire :

```javascript
console.log(req.body);
```

Mais dans une vraie application, il faut éviter d’afficher des informations sensibles dans le terminal.

Par exemple, il ne faut pas afficher :

- mots de passe ;
- jetons d’authentification ;
- informations personnelles sensibles ;
- données confidentielles.

---

## Répondre clairement au client

Un serveur doit toujours répondre.

À éviter :

```javascript
app.post("/api/contact", function (req, res) {
  console.log(req.body);
});
```

Ce code reçoit la requête, mais ne répond pas.

Correction :

```javascript
app.post("/api/contact", function (req, res) {
  console.log(req.body);

  res.json({
    message: "Données reçues.",
  });
});
```

---

## Utiliser les bons statuts HTTP

Quelques statuts utiles :

```txt
200 → succès
201 → ressource créée
400 → erreur dans la requête du client
404 → route introuvable
500 → erreur serveur
```

Exemple :

```javascript
return res.status(400).json({
  message: "Le formulaire est incomplet.",
});
```

# 44. Ce qu’il faut absolument retenir

## Côté navigateur

Pour envoyer un formulaire avec `fetch`, on utilise souvent cette structure :

```javascript
form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  const response = await fetch("/api/contact", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(donnees),
  });

  const data = await response.json();
});
```

---

## Côté serveur

Pour recevoir du JSON avec Express :

```javascript
app.use(express.json());

app.post("/api/contact", function (req, res) {
  console.log(req.body);

  res.json({
    message: "Données reçues.",
  });
});
```

---

## Le lien entre les deux

```txt
fetch("/api/contact", { method: "POST" })
```

doit correspondre à :

```javascript
app.post("/api/contact", ...)
```

L’adresse et la méthode doivent être cohérentes.

---

## `Content-Type` et middleware

Si le client envoie :

```javascript
"Content-Type": "application/json"
```

alors le serveur doit utiliser :

```javascript
app.use(express.json());
```

Si le formulaire est soumis classiquement sans `fetch`, le serveur utilise plutôt :

```javascript
app.use(express.urlencoded({ extended: true }));
```

# Conclusion

L’envoi avec `fetch` et la réception avec Express permettent de comprendre le fonctionnement réel d’une application web moderne.

Le navigateur prépare les données du formulaire, les transforme en JSON et les envoie avec une requête HTTP. Le serveur Express reçoit cette requête, lit les données avec un middleware, les valide, puis renvoie une réponse au navigateur.

La logique à retenir est la suivante :

```txt
Formulaire HTML
   ↓
JavaScript récupère les données
   ↓
JSON.stringify transforme l’objet en JSON
   ↓
fetch envoie une requête POST
   ↓
Express reçoit la requête
   ↓
express.json() lit le body
   ↓
req.body contient les données
   ↓
Le serveur valide et répond
   ↓
Le navigateur affiche la réponse
```

À partir de ce point, vous pouvez commencer à relier le front-end et le back-end. Ils ne construisent plus seulement une page interactive : ils construisent une vraie communication entre le navigateur et le serveur.
