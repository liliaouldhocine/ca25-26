# Support de cours

# Accessibilité, bonnes pratiques, sécurité et erreurs fréquentes dans les formulaires

## Présentation générale

Dans les cours précédents, nous avons appris à construire un formulaire, à récupérer ses données avec JavaScript, à valider les champs, à envoyer les données avec `fetch` et à les recevoir côté serveur avec Express.

Nous allons maintenant aborder deux dimensions essentielles d’un formulaire professionnel :

```txt
1. L’accessibilité et les bonnes pratiques
2. La sécurité et les erreurs fréquentes
```

Un formulaire ne doit pas seulement fonctionner. Il doit être utilisable par le plus grand nombre de personnes possible, clair pour l’utilisateur, robuste pour le développeur et prudent du point de vue de la sécurité.

Les recommandations du W3C/WAI rappellent que les formulaires accessibles reposent notamment sur des labels, des instructions, des groupes de champs et des messages d’erreur compréhensibles. Les critères WCAG 2.2 incluent aussi une section importante sur l’aide à la saisie, avec l’identification des erreurs, les labels ou instructions, les suggestions de correction et la prévention des erreurs. ([w3.org][1])

---

# Partie 1 — Accessibilité et bonnes pratiques

## 1. Qu’est-ce que l’accessibilité ?

L’accessibilité consiste à concevoir des interfaces utilisables par des personnes ayant des besoins variés.

Cela peut concerner :

- une personne qui utilise un lecteur d’écran ;
- une personne qui navigue uniquement au clavier ;
- une personne ayant une difficulté visuelle ;
- une personne ayant une difficulté motrice ;
- une personne ayant une difficulté cognitive ;
- une personne fatiguée, pressée ou peu à l’aise avec l’informatique ;
- une personne qui utilise un téléphone, une tablette ou un petit écran.

L’accessibilité ne concerne donc pas seulement une minorité d’utilisateurs. Elle améliore l’expérience de tout le monde.

Un formulaire accessible est généralement :

```txt
clair
prévisible
bien structuré
utilisable au clavier
lisible
compréhensible
tolérant aux erreurs
```

---

## 2. Pourquoi l’accessibilité est-elle importante dans les formulaires ?

Les formulaires demandent une action à l’utilisateur.

L’utilisateur doit comprendre :

```txt
ce qu’on lui demande
où écrire
quels champs sont obligatoires
quel format est attendu
ce qui ne va pas en cas d’erreur
comment corriger l’erreur
comment soumettre le formulaire
```

Un formulaire mal conçu peut empêcher certaines personnes de terminer une inscription, une demande, un achat ou une communication.

L’objectif n’est pas seulement d’avoir un formulaire joli. L’objectif est d’avoir un formulaire utilisable.

---

# 3. Utiliser des labels visibles et associés aux champs

La première bonne pratique est d’utiliser un vrai `<label>` pour chaque champ important.

## Mauvaise pratique

```html
<input type="email" placeholder="Email" />
```

Ce champ peut sembler compréhensible, mais il n’a pas de label réel. Le texte disparaît lorsque l’utilisateur commence à écrire.

## Bonne pratique

```html
<label for="email">Adresse courriel</label>
<input type="email" id="email" name="email" />
```

Le label est visible, clair et associé au champ.

Le lien se fait ainsi :

```html
<label for="email">Adresse courriel</label> <input id="email" />
```

Le `for` du label correspond au `id` du champ.

Les recommandations WAI insistent sur l’importance de labels correctement associés aux champs afin de décrire clairement le rôle de chaque contrôle de formulaire. ([w3.org][2])

---

# 4. Ne pas remplacer le label par le placeholder

Le placeholder peut donner un exemple, mais il ne doit pas remplacer le label.

## Mauvaise pratique

```html
<input type="text" placeholder="Nom complet" />
```

## Meilleure pratique

```html
<label for="nom">Nom complet</label>
<input type="text" id="nom" name="nom" placeholder="Ex. Sara Dupont" />
```

Le label indique ce que représente le champ.
Le placeholder donne seulement un exemple.

Le WAI recommande de fournir des instructions qui aident à comprendre les champs, les formats attendus et les entrées obligatoires ou facultatives. ([w3.org][3])

---

# 5. Utiliser des instructions claires

Certains champs ont besoin d’une explication.

Exemple :

```html
<label for="password">Mot de passe</label>
<p id="passwordHelp">Le mot de passe doit contenir au moins 8 caractères.</p>
<input
  type="password"
  id="password"
  name="password"
  aria-describedby="passwordHelp"
/>
```

Ici, le champ est lié à une instruction grâce à :

```html
aria-describedby="passwordHelp"
```

Cela permet d’associer le champ à un texte d’aide.

L’objectif est d’éviter que l’utilisateur devine ce qu’il doit faire.

---

# 6. Indiquer clairement les champs obligatoires

Un champ obligatoire doit être annoncé clairement.

## Exemple

```html
<label for="email">Adresse courriel <span aria-hidden="true">*</span></label>
<p id="emailHelp">Champ obligatoire.</p>
<input
  type="email"
  id="email"
  name="email"
  required
  aria-describedby="emailHelp"
/>
```

Le symbole `*` peut être utile visuellement, mais il ne doit pas être la seule indication. Il faut aussi expliquer ce qu’il signifie.

## Meilleure approche globale

```html
<p>Les champs marqués d’un astérisque (*) sont obligatoires.</p>
```

Puis :

```html
<label for="email">Adresse courriel *</label>
<input type="email" id="email" name="email" required />
```

---

# 7. Regrouper les champs liés avec `fieldset` et `legend`

Lorsqu’un groupe de champs appartient à une même question, il faut le regrouper.

C’est particulièrement important pour :

- les boutons radio ;
- les cases à cocher ;
- les informations personnelles ;
- les sections d’un formulaire long.

## Exemple

```html
<fieldset>
  <legend>Mode de participation</legend>

  <label>
    <input type="radio" name="mode" value="presentiel" />
    Présentiel
  </label>

  <label>
    <input type="radio" name="mode" value="distance" />
    À distance
  </label>

  <label>
    <input type="radio" name="mode" value="hybride" />
    Hybride
  </label>
</fieldset>
```

Le `<legend>` donne le sens du groupe.

Sans `<legend>`, les options peuvent être comprises séparément, mais l’utilisateur ne sait pas toujours à quelle question elles répondent.

Le WAI recommande de regrouper les contrôles liés pour rendre les formulaires plus compréhensibles, notamment avec `fieldset` et `legend` pour les groupes de radio buttons, checkbox ou champs liés. ([w3.org][4])

---

# 8. Prévoir une navigation au clavier

Un formulaire accessible doit être utilisable sans souris.

L’utilisateur doit pouvoir :

```txt
passer d’un champ à l’autre avec Tab
activer une checkbox avec Espace
choisir une option radio avec le clavier
soumettre le formulaire avec Entrée
voir clairement quel champ est actif
```

Il ne faut pas retirer l’indication de focus sans la remplacer.

## Mauvaise pratique

```css
input:focus {
  outline: none;
}
```

Cela peut rendre la navigation au clavier difficile.

## Meilleure pratique

```css
input:focus,
textarea:focus,
select:focus {
  outline: 2px solid #333;
  outline-offset: 2px;
}
```

Le champ actif est visible.

---

# 9. Utiliser un ordre logique dans le HTML

L’ordre des champs dans le HTML doit correspondre à l’ordre visuel.

## Exemple logique

```html
<label for="prenom">Prénom</label>
<input type="text" id="prenom" name="prenom" />

<label for="nom">Nom</label>
<input type="text" id="nom" name="nom" />

<label for="email">Adresse courriel</label>
<input type="email" id="email" name="email" />
```

L’utilisateur parcourt le formulaire dans un ordre naturel :

```txt
Prénom → Nom → Email
```

Il faut éviter de créer une apparence visuelle qui contredit l’ordre réel du HTML.

---

# 10. Éviter les formulaires trop longs et mal divisés

Un formulaire long peut être intimidant.

On peut améliorer l’expérience en divisant le formulaire en sections :

```html
<form>
  <fieldset>
    <legend>Informations personnelles</legend>
    ...
  </fieldset>

  <fieldset>
    <legend>Préférences</legend>
    ...
  </fieldset>

  <fieldset>
    <legend>Confirmation</legend>
    ...
  </fieldset>
</form>
```

Cela aide l’utilisateur à comprendre la progression.

Pour un très grand formulaire, on peut aussi envisager un formulaire en plusieurs étapes, à condition de bien indiquer où l’utilisateur se trouve.

---

# 11. Écrire des messages d’erreur accessibles

Un message d’erreur doit être :

```txt
clair
proche du champ
compréhensible
en texte
utile pour corriger l’erreur
```

## Mauvaise pratique

```txt
Erreur.
```

## Meilleure pratique

```txt
Veuillez entrer une adresse courriel valide.
```

## Exemple HTML

```html
<label for="email">Adresse courriel</label>
<input type="email" id="email" name="email" aria-describedby="emailError" />
<p id="emailError" class="error"></p>
```

## JavaScript

```javascript
if (email === "") {
  emailError.textContent = "L’adresse courriel est obligatoire.";
}
```

WCAG 2.2 indique que lorsqu’une erreur de saisie est détectée automatiquement, l’élément en erreur doit être identifié et l’erreur doit être décrite à l’utilisateur en texte. ([w3.org][5])

---

# 12. Ne pas compter uniquement sur la couleur

Un champ rouge peut indiquer une erreur, mais la couleur ne doit pas être le seul indice.

## Insuffisant

```css
.input-error {
  border: 2px solid red;
}
```

Si l’utilisateur ne distingue pas bien les couleurs, il peut ne pas comprendre l’erreur.

## Meilleur exemple

```html
<input class="input-error" aria-describedby="emailError" />
<p id="emailError">L’adresse courriel est obligatoire.</p>
```

On combine :

```txt
un message textuel
un style visuel
un lien entre le champ et le message
```

---

# 13. Utiliser `aria-invalid` lorsque le champ est invalide

Lorsqu’un champ contient une erreur, on peut ajouter :

```html
aria-invalid="true"
```

Exemple :

```javascript
if (email === "") {
  emailInput.setAttribute("aria-invalid", "true");
  emailError.textContent = "L’adresse courriel est obligatoire.";
} else {
  emailInput.removeAttribute("aria-invalid");
  emailError.textContent = "";
}
```

Cela donne une information supplémentaire aux technologies d’assistance.

---

# 14. Utiliser `aria-live` pour annoncer un résultat dynamique

Quand un message apparaît après une action, il peut être utile de le rendre annonçable.

Exemple :

```html
<p id="resultat" aria-live="polite"></p>
```

Puis :

```javascript
resultat.textContent = "Votre message a bien été envoyé.";
```

`aria-live="polite"` indique que le message peut être annoncé sans interrompre brutalement l’utilisateur.

Cette technique est utile pour :

- message de succès ;
- message d’erreur général ;
- résultat d’envoi ;
- confirmation dynamique.

---

# 15. Utiliser `autocomplete` quand c’est pertinent

L’attribut `autocomplete` aide le navigateur à proposer des valeurs connues.

Exemples :

```html
<label for="prenom">Prénom</label>
<input type="text" id="prenom" name="prenom" autocomplete="given-name" />

<label for="email">Adresse courriel</label>
<input type="email" id="email" name="email" autocomplete="email" />
```

Cela peut améliorer l’expérience utilisateur, surtout sur mobile.

---

# 16. Prévoir des champs adaptés au type de donnée

Utiliser le bon type de champ aide le navigateur et l’utilisateur.

## Exemples

```html
<input type="email" />
<input type="tel" />
<input type="number" />
<input type="date" />
<input type="url" />
<input type="password" />
```

Ces types peuvent améliorer :

- le clavier affiché sur mobile ;
- la validation de base ;
- la compréhension du champ ;
- la qualité des données.

---

# 17. Soigner les boutons

Les boutons doivent être explicites.

## Moins clair

```html
<button type="submit">OK</button>
```

## Plus clair

```html
<button type="submit">Envoyer le message</button>
```

ou :

```html
<button type="submit">Créer le compte</button>
```

Le texte du bouton doit décrire l’action réelle.

---

# 18. Désactiver un bouton avec prudence

On peut désactiver un bouton tant que le formulaire n’est pas prêt.

```html
<button type="submit" id="submitButton" disabled>Envoyer</button>
```

Mais il faut éviter de bloquer l’utilisateur sans explication.

## Meilleure pratique

```html
<p id="conditionsHelp">
  Vous devez accepter les conditions avant d’envoyer le formulaire.
</p>

<button type="submit" id="submitButton" disabled>Envoyer</button>
```

Le bouton désactivé doit avoir une raison compréhensible.

# 19. Exemple complet accessible

```html
<form id="contactForm" novalidate>
  <p>Les champs marqués d’un astérisque (*) sont obligatoires.</p>

  <div>
    <label for="nom">Nom complet *</label>
    <input
      type="text"
      id="nom"
      name="nom"
      autocomplete="name"
      required
      aria-describedby="nomError"
    />
    <p id="nomError" class="error"></p>
  </div>

  <div>
    <label for="email">Adresse courriel *</label>
    <p id="emailHelp">Exemple : nom@exemple.com</p>
    <input
      type="email"
      id="email"
      name="email"
      autocomplete="email"
      required
      aria-describedby="emailHelp emailError"
    />
    <p id="emailError" class="error"></p>
  </div>

  <fieldset>
    <legend>Mode de contact préféré</legend>

    <label>
      <input type="radio" name="modeContact" value="email" />
      Courriel
    </label>

    <label>
      <input type="radio" name="modeContact" value="telephone" />
      Téléphone
    </label>
  </fieldset>

  <div>
    <label for="message">Message *</label>
    <textarea
      id="message"
      name="message"
      required
      aria-describedby="messageError"
    ></textarea>
    <p id="messageError" class="error"></p>
  </div>

  <button type="submit">Envoyer le message</button>

  <p id="resultat" aria-live="polite"></p>
</form>
```

## CSS

```css
.error {
  color: #b00020;
  font-size: 0.9rem;
}

input:focus,
textarea:focus,
select:focus,
button:focus {
  outline: 2px solid #333;
  outline-offset: 2px;
}

.input-error {
  border: 2px solid #b00020;
}
```

## JavaScript

```javascript
const form = document.querySelector("#contactForm");

const nomInput = document.querySelector("#nom");
const emailInput = document.querySelector("#email");
const messageInput = document.querySelector("#message");

const nomError = document.querySelector("#nomError");
const emailError = document.querySelector("#emailError");
const messageError = document.querySelector("#messageError");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", function (event) {
  event.preventDefault();

  nomError.textContent = "";
  emailError.textContent = "";
  messageError.textContent = "";
  resultat.textContent = "";

  nomInput.classList.remove("input-error");
  emailInput.classList.remove("input-error");
  messageInput.classList.remove("input-error");

  nomInput.removeAttribute("aria-invalid");
  emailInput.removeAttribute("aria-invalid");
  messageInput.removeAttribute("aria-invalid");

  const nom = nomInput.value.trim();
  const email = emailInput.value.trim();
  const message = messageInput.value.trim();

  let formulaireValide = true;

  if (nom === "") {
    nomError.textContent = "Le nom complet est obligatoire.";
    nomInput.classList.add("input-error");
    nomInput.setAttribute("aria-invalid", "true");
    formulaireValide = false;
  }

  if (email === "") {
    emailError.textContent = "L’adresse courriel est obligatoire.";
    emailInput.classList.add("input-error");
    emailInput.setAttribute("aria-invalid", "true");
    formulaireValide = false;
  }

  if (message === "") {
    messageError.textContent = "Le message est obligatoire.";
    messageInput.classList.add("input-error");
    messageInput.setAttribute("aria-invalid", "true");
    formulaireValide = false;
  }

  if (!formulaireValide) {
    resultat.textContent = "Veuillez corriger les champs indiqués.";
    return;
  }

  resultat.textContent = "Le formulaire est valide.";
});
```

# 20. Bonnes pratiques générales à retenir

Un formulaire professionnel devrait respecter ces principes :

```txt
des labels visibles
des champs bien nommés
des instructions claires
des messages d’erreur précis
un ordre logique
une navigation clavier possible
un focus visible
des groupes de champs structurés
des boutons explicites
une validation côté client pour aider l’utilisateur
une validation côté serveur pour protéger l’application
```

# Partie 2 — Sécurité et erreurs fréquentes

## 21. Pourquoi parler de sécurité dans un cours sur les formulaires ?

Un formulaire est une porte d’entrée.

Chaque champ permet à l’utilisateur d’envoyer des informations à l’application.

Cela signifie que le serveur peut recevoir :

```txt
des données correctes
des données incomplètes
des données mal formatées
des données trop longues
des données inattendues
des données potentiellement dangereuses
```

Un formulaire doit donc être pensé avec prudence.

Le principe de base est simple :

```txt
Ne jamais faire confiance aveuglément aux données reçues.
```

OWASP recommande de valider les entrées pour s’assurer que seules des données correctement formées entrent dans le flux de traitement d’une application, idéalement dès qu’elles sont reçues depuis une source externe. ([OWASP Cheat Sheet Series][6])

---

# 22. Validation côté client et validation côté serveur

## Validation côté client

La validation côté client se fait dans le navigateur.

Elle sert surtout à :

```txt
aider l’utilisateur
afficher rapidement les erreurs
éviter des requêtes inutiles
améliorer l’expérience utilisateur
```

Exemple :

```javascript
if (email === "") {
  emailError.textContent = "L’adresse courriel est obligatoire.";
}
```

## Validation côté serveur

La validation côté serveur se fait dans Express, Node.js ou une autre technologie back-end.

Elle sert à :

```txt
protéger l’application
vérifier les données réellement reçues
éviter les données incohérentes
refuser les entrées invalides
prévenir des erreurs dans la base de données
```

Exemple :

```javascript
app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();
  const email = req.body.email?.trim();
  const message = req.body.message?.trim();

  if (!nom || !email || !message) {
    return res.status(400).json({
      message: "Tous les champs sont obligatoires.",
    });
  }

  res.json({
    message: "Message reçu.",
  });
});
```

La validation côté client ne remplace jamais la validation côté serveur.

---

# 23. Pourquoi la validation côté client ne suffit pas ?

Un utilisateur peut :

- désactiver JavaScript ;
- modifier le HTML dans le navigateur ;
- envoyer une requête directement à l’API ;
- contourner les attributs `required`, `minlength`, `type="email"` ;
- envoyer des données dans un format inattendu.

Donc même si le formulaire semble bien validé dans la page, le serveur doit refaire ses propres vérifications.

OWASP rappelle aussi que les paramètres reçus par une API ne doivent pas être considérés comme fiables et doivent être validés selon le type, le format, la longueur ou l’intervalle attendu. ([OWASP Cheat Sheet Series][7])

---

# 24. Définir ce que l’on accepte

Une bonne validation commence par une question :

```txt
Qu’est-ce qui est acceptable ?
```

Exemples :

```txt
Le nom doit être une chaîne de 2 à 80 caractères.
L’email doit ressembler à une adresse courriel.
Le message doit contenir entre 10 et 1000 caractères.
L’âge doit être un nombre entre 12 et 120.
Le niveau doit être uniquement : debutant, intermediaire ou avance.
```

## Exemple côté serveur

```javascript
const niveauxAutorises = ["debutant", "intermediaire", "avance"];

if (!niveauxAutorises.includes(niveau)) {
  return res.status(400).json({
    message: "Le niveau choisi est invalide.",
  });
}
```

Cette approche est plus sécuritaire que d’accepter n’importe quelle valeur.

---

# 25. Contrôler la longueur des champs

Un champ texte ne devrait pas accepter une longueur illimitée.

## Exemple

```javascript
if (message.length > 1000) {
  return res.status(400).json({
    message: "Le message ne doit pas dépasser 1000 caractères.",
  });
}
```

Pourquoi ?

Parce que les textes trop longs peuvent :

- ralentir le traitement ;
- créer des problèmes d’affichage ;
- remplir inutilement la base de données ;
- provoquer des comportements inattendus.

---

# 26. Contrôler le type des données

Même si un champ HTML utilise :

```html
<input type="number" />
```

le serveur doit quand même vérifier que la donnée reçue est bien utilisable comme nombre.

## Exemple

```javascript
const age = Number(req.body.age);

if (Number.isNaN(age)) {
  return res.status(400).json({
    message: "L’âge doit être un nombre.",
  });
}

if (age < 12 || age > 120) {
  return res.status(400).json({
    message: "L’âge doit être compris entre 12 et 120.",
  });
}
```

---

# 27. Faire attention à `innerHTML`

Une erreur fréquente côté client consiste à afficher une donnée utilisateur avec `innerHTML`.

## À éviter

```javascript
resultat.innerHTML = nomInput.value;
```

Cela peut être dangereux si la donnée contient du contenu interprété comme du HTML.

## Meilleure pratique

```javascript
resultat.textContent = nomInput.value;
```

`textContent` affiche le contenu comme du texte.

Dans un cours débutant, la règle simple est :

```txt
Pour afficher une donnée écrite par l’utilisateur, utiliser textContent.
```

---

# 28. Comprendre le risque XSS sans entrer dans l’exploitation

Le XSS, ou Cross-Site Scripting, est un risque lié à l’insertion de contenu non fiable dans une page web.

Dans un contexte de formulaire, le risque apparaît lorsqu’une application affiche directement une donnée utilisateur comme si elle était du HTML de confiance.

Exemple risqué :

```javascript
zoneResultat.innerHTML = messageUtilisateur;
```

Exemple plus sûr :

```javascript
zoneResultat.textContent = messageUtilisateur;
```

OWASP recommande notamment l’encodage de sortie comme mesure importante contre les risques XSS, c’est-à-dire traiter les données non fiables de manière adaptée avant de les insérer dans une page. ([OWASP Cheat Sheet Series][8])

---

# 29. Ne pas afficher d’informations sensibles dans la console

Pendant l’apprentissage, on utilise souvent :

```javascript
console.log(req.body);
```

C’est utile pour comprendre.

Mais dans une vraie application, il faut éviter d’afficher :

```txt
mots de passe
jetons de session
informations bancaires
données personnelles sensibles
numéros d’identification
informations médicales
```

## À éviter

```javascript
console.log("Mot de passe reçu :", password);
```

## Meilleur principe

```txt
Ne jamais journaliser les secrets.
```

OWASP signale que les mauvaises pratiques de journalisation peuvent mener à des fuites d’informations sensibles et à d’autres risques liés aux logs. ([owasp.org][9])

---

# 30. Ne pas envoyer de mot de passe avec GET

Un formulaire de connexion ne doit pas utiliser GET.

## À éviter

```html
<form action="/login" method="GET">
  <input type="email" name="email" />
  <input type="password" name="password" />
  <button type="submit">Connexion</button>
</form>
```

Avec GET, les données peuvent apparaître dans l’URL.

## Meilleure pratique

```html
<form action="/login" method="POST">
  <input type="email" name="email" />
  <input type="password" name="password" />
  <button type="submit">Connexion</button>
</form>
```

Et dans une vraie application, il faut utiliser HTTPS.

---

# 31. POST ne signifie pas automatiquement sécurisé

POST évite d’afficher les données dans l’URL, mais il ne chiffre pas automatiquement la communication.

À retenir :

```txt
GET place les données dans l’URL.
POST place les données dans le corps de la requête.
HTTPS protège la communication entre le navigateur et le serveur.
```

Donc, pour des données sensibles :

```txt
POST + HTTPS + validation serveur + bonnes pratiques de stockage
```

---

# 32. Ne jamais stocker un mot de passe en clair

Dans une vraie application, un mot de passe ne doit pas être stocké directement en clair.

## À éviter

```javascript
const utilisateur = {
  email: email,
  password: password,
};
```

Ce type d’exemple peut être acceptable pour expliquer temporairement la structure des données, mais il ne doit pas être présenté comme une bonne pratique de stockage.

## Principe professionnel

```txt
Un mot de passe doit être traité avec des mécanismes spécialisés côté serveur.
```

Pour un cours débutant, il suffit de retenir :

```txt
On ne stocke pas les mots de passe en clair.
On ne les affiche pas dans la console.
On ne les envoie pas avec GET.
```

---

# 33. Limiter ce que le serveur accepte

Le serveur ne doit pas accepter automatiquement toutes les propriétés reçues.

Exemple d’objet reçu :

```javascript
{
  nom: "Sara",
  email: "sara@email.com",
  role: "admin"
}
```

Même si le formulaire ne contenait pas de champ `role`, un client peut envoyer une propriété inattendue.

## Mauvaise approche

```javascript
const nouvelUtilisateur = req.body;
```

## Meilleure approche

```javascript
const nouvelUtilisateur = {
  nom: req.body.nom?.trim(),
  email: req.body.email?.trim(),
};
```

On choisit explicitement les champs acceptés.

---

# 34. Utiliser une liste de valeurs autorisées

Pour les champs comme `niveau`, `role`, `categorie`, `mode`, il faut éviter d’accepter n’importe quelle valeur.

## Exemple

```javascript
const niveauxAutorises = ["debutant", "intermediaire", "avance"];

if (!niveauxAutorises.includes(niveau)) {
  return res.status(400).json({
    message: "Niveau invalide.",
  });
}
```

Cette pratique évite d’enregistrer des valeurs inattendues.

---

# 35. Ne pas exposer trop de détails dans les messages d’erreur

Un message d’erreur doit être utile, mais pas trop détaillé.

## Exemple acceptable

```json
{
  "message": "Les informations envoyées sont invalides."
}
```

## À éviter dans une vraie application

```json
{
  "message": "Erreur SQL dans la table users à la ligne..."
}
```

Les détails techniques internes ne doivent pas être exposés à l’utilisateur.

Ils peuvent être utiles côté serveur dans les logs, mais pas dans la réponse publique.

---

# 36. Répondre avec des statuts HTTP appropriés

Les statuts HTTP aident le client à comprendre le résultat.

Exemples :

```txt
200 → succès
201 → ressource créée
400 → requête invalide
401 → non authentifié
403 → interdit
404 → route introuvable
500 → erreur serveur
```

## Exemple

```javascript
return res.status(400).json({
  message: "Tous les champs sont obligatoires.",
});
```

Cela indique que la requête du client n’est pas correcte.

---

# 37. Toujours répondre au client

Une route Express doit toujours envoyer une réponse.

## Erreur fréquente

```javascript
app.post("/api/contact", function (req, res) {
  console.log(req.body);
});
```

Le client attend une réponse qui ne vient jamais.

## Correction

```javascript
app.post("/api/contact", function (req, res) {
  console.log(req.body);

  res.json({
    message: "Données reçues.",
  });
});
```

---

# 38. Éviter les doubles réponses

Une route ne doit pas envoyer deux réponses pour la même requête.

## Erreur

```javascript
if (!email) {
  res.status(400).json({
    message: "Email obligatoire.",
  });
}

res.json({
  message: "Succès.",
});
```

Même si l’email est absent, le code continue et essaie d’envoyer une deuxième réponse.

## Correction

```javascript
if (!email) {
  return res.status(400).json({
    message: "Email obligatoire.",
  });
}

res.json({
  message: "Succès.",
});
```

Le `return` arrête la fonction.

---

# 39. Protéger contre les requêtes trop volumineuses

Un serveur ne devrait pas accepter des corps de requêtes illimités.

Dans Express, on peut limiter la taille du JSON reçu :

```javascript
app.use(express.json({ limit: "10kb" }));
```

Cela peut aider à éviter que des données trop volumineuses soient acceptées inutilement.

OWASP recommande de définir une taille appropriée pour les requêtes et de refuser celles qui dépassent cette limite. ([OWASP Cheat Sheet Series][7])

---

# 40. Faire attention à CORS

CORS concerne les communications entre origines différentes.

Exemple :

```txt
Client : http://localhost:5173
Serveur : http://localhost:3000
```

Dans ce cas, le navigateur peut bloquer certaines requêtes si le serveur ne les autorise pas.

Dans un premier projet pédagogique, on peut simplifier en servant le client directement avec Express :

```javascript
app.use(express.static("public"));
```

Ainsi, la page et l’API utilisent la même origine :

```txt
http://localhost:3000
```

# 41. Erreurs fréquentes côté client

## Erreur 1 : oublier `preventDefault()`

```javascript
form.addEventListener("submit", async function (event) {
  const formData = new FormData(form);
});
```

Correction :

```javascript
form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
});
```

---

## Erreur 2 : oublier `JSON.stringify()`

Erreur :

```javascript
body: donnees;
```

Correction :

```javascript
body: JSON.stringify(donnees);
```

---

## Erreur 3 : oublier le header JSON

Erreur :

```javascript
fetch("/api/contact", {
  method: "POST",
  body: JSON.stringify(donnees),
});
```

Correction :

```javascript
fetch("/api/contact", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(donnees),
});
```

---

## Erreur 4 : utiliser une mauvaise URL

Client :

```javascript
fetch("/api/contact");
```

Serveur :

```javascript
app.post("/contact", ...)
```

Correction : les deux doivent correspondre.

```javascript
fetch("/api/contact");
```

et :

```javascript
app.post("/api/contact", ...)
```

---

## Erreur 5 : lire du JSON alors que le serveur renvoie du texte

Client :

```javascript
const data = await response.json();
```

Serveur :

```javascript
res.send("Données reçues.");
```

Correction possible côté serveur :

```javascript
res.json({
  message: "Données reçues.",
});
```

ou côté client :

```javascript
const texte = await response.text();
```

# 42. Erreurs fréquentes côté serveur

## Erreur 1 : oublier `express.json()`

```javascript
app.post("/api/contact", function (req, res) {
  console.log(req.body);
});
```

Correction :

```javascript
app.use(express.json());

app.post("/api/contact", function (req, res) {
  console.log(req.body);
});
```

---

## Erreur 2 : ne pas valider `req.body`

Erreur :

```javascript
app.post("/api/contact", function (req, res) {
  res.json({
    message: "Données acceptées.",
  });
});
```

Correction :

```javascript
app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();

  if (!nom) {
    return res.status(400).json({
      message: "Le nom est obligatoire.",
    });
  }

  res.json({
    message: "Données acceptées.",
  });
});
```

---

## Erreur 3 : faire confiance aux données cachées

Même si un champ est caché dans le HTML, il peut être envoyé.

Le serveur doit donc vérifier ce qu’il reçoit.

---

## Erreur 4 : afficher des informations sensibles dans les logs

Erreur :

```javascript
console.log(req.body);
```

Cela peut être acceptable pendant l’apprentissage, mais il faut éviter de le faire avec des données sensibles.

Meilleure pratique :

```javascript
console.log("Formulaire reçu.");
```

---

## Erreur 5 : ne pas gérer les erreurs

Une route doit prévoir les cas où les données sont invalides.

```javascript
if (!email) {
  return res.status(400).json({
    message: "L’adresse courriel est obligatoire.",
  });
}
```

# 43. Exemple complet sécurisé de base

## `server.js`

```javascript
import express from "express";

const app = express();
const PORT = 3000;

app.use(express.static("public"));
app.use(express.json({ limit: "10kb" }));

app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();
  const email = req.body.email?.trim();
  const message = req.body.message?.trim();

  if (!nom || !email || !message) {
    return res.status(400).json({
      message: "Tous les champs sont obligatoires.",
    });
  }

  if (nom.length < 2 || nom.length > 80) {
    return res.status(400).json({
      message: "Le nom doit contenir entre 2 et 80 caractères.",
    });
  }

  if (message.length < 10 || message.length > 1000) {
    return res.status(400).json({
      message: "Le message doit contenir entre 10 et 1000 caractères.",
    });
  }

  console.log("Message de contact reçu.");

  res.status(200).json({
    message: "Votre message a bien été reçu.",
  });
});

app.listen(PORT, function () {
  console.log(`Serveur lancé sur http://localhost:${PORT}`);
});
```

## `public/script.js`

```javascript
const form = document.querySelector("#contactForm");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  resultat.textContent = "";

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    if (!response.ok) {
      resultat.textContent = data.message;
      return;
    }

    resultat.textContent = data.message;
    form.reset();
  } catch (error) {
    resultat.textContent = "Impossible de contacter le serveur.";
    console.error("Erreur de communication avec le serveur.");
  }
});
```

# 44. Liste de vérification — accessibilité

Avant de considérer un formulaire comme terminé, vérifier :

| Point à vérifier                                                       | Oui / Non |
| ---------------------------------------------------------------------- | --------- |
| Chaque champ important possède un label visible                        |           |
| Les labels sont liés aux champs avec `for` et `id`                     |           |
| Le placeholder ne remplace pas le label                                |           |
| Les champs obligatoires sont clairement indiqués                       |           |
| Les messages d’erreur sont textuels et compréhensibles                 |           |
| Les messages d’erreur sont proches des champs concernés                |           |
| Les groupes de radio/checkbox utilisent `fieldset` et `legend`         |           |
| Le formulaire est utilisable au clavier                                |           |
| Le focus est visible                                                   |           |
| Les boutons ont des textes explicites                                  |           |
| Les instructions importantes sont associées aux champs                 |           |
| Les messages dynamiques importants utilisent `aria-live` si nécessaire |           |
| La couleur n’est pas le seul moyen d’indiquer une erreur               |           |

---

# 45. Liste de vérification — sécurité

Avant de considérer un formulaire comme prêt côté serveur, vérifier :

| Point à vérifier                                                    | Oui / Non |
| ------------------------------------------------------------------- | --------- |
| Les données sont validées côté client pour l’expérience utilisateur |           |
| Les données sont validées côté serveur pour la sécurité             |           |
| Le serveur ne fait pas confiance directement à `req.body`           |           |
| Les champs attendus sont sélectionnés explicitement                 |           |
| Les valeurs autorisées sont vérifiées                               |           |
| Les longueurs maximales sont contrôlées                             |           |
| Les champs numériques sont convertis et vérifiés                    |           |
| Les données utilisateur ne sont pas affichées avec `innerHTML`      |           |
| Les informations sensibles ne sont pas affichées dans les logs      |           |
| Les mots de passe ne sont pas envoyés avec GET                      |           |
| Les erreurs publiques ne révèlent pas de détails internes           |           |
| Le serveur répond toujours au client                                |           |
| Les réponses d’erreur utilisent des statuts appropriés              |           |
| Les corps de requêtes trop volumineux sont limités                  |           |

# 46. Ce qu’il faut absolument retenir

## Accessibilité

Un formulaire accessible doit être compréhensible et utilisable par le plus grand nombre possible de personnes.

Les éléments essentiels sont :

```txt
labels visibles
instructions claires
messages d’erreur textuels
groupes bien structurés
navigation clavier
focus visible
boutons explicites
```

---

## Sécurité

Un formulaire est une porte d’entrée vers l’application.

Le serveur doit toujours vérifier ce qu’il reçoit.

Les réflexes essentiels sont :

```txt
ne jamais faire confiance uniquement au client
valider côté serveur
contrôler les types, formats et longueurs
ne pas exposer les détails internes
ne pas afficher les données sensibles
éviter innerHTML avec les données utilisateur
utiliser POST et HTTPS pour les données sensibles
```

---

# Conclusion

Un bon formulaire n’est pas seulement un formulaire qui fonctionne.

C’est un formulaire :

```txt
clair pour l’utilisateur
accessible au clavier
compréhensible avec un lecteur d’écran
prévisible
tolérant aux erreurs
validé côté client
validé côté serveur
prudent avec les données reçues
```

L’accessibilité améliore l’expérience de tous les utilisateurs.
La sécurité protège l’application et les données.
Les bonnes pratiques rendent le code plus fiable, plus professionnel et plus facile à maintenir.

La logique finale à retenir est la suivante :

```txt
Un formulaire demande une information.
L’utilisateur doit comprendre quoi entrer.
Le navigateur peut aider.
JavaScript peut guider.
Le serveur doit vérifier.
L’application doit répondre clairement.
Les données doivent être traitées avec prudence.
```

À partir de ce point, vous ne créez plus seulement des formulaires fonctionnels. Vous commencez à créer des formulaires responsables, utilisables et plus proches des attentes d’une vraie application web.
