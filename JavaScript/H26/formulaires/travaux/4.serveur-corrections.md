# Corrigé de l’exercice 1 — Comprendre la structure d’un envoi avec `fetch`

```javascript
const response = await fetch("/api/contact", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify(donnees),
});

// 1. "/api/contact" correspond à l’adresse de la route serveur qui recevra la requête.

// 2. method: "POST" indique que l’on envoie des données au serveur.

// 3. L’objet headers permet d’envoyer des informations supplémentaires avec la requête.

// 4. "Content-Type": "application/json" indique que les données envoyées sont au format JSON.

// 5. body contient les données envoyées dans le corps de la requête.

// 6. JSON.stringify(donnees) transforme l’objet JavaScript en chaîne JSON,
// car une requête HTTP envoie du texte, pas directement un objet JavaScript.

// 7. response contient la réponse du serveur.
// Elle permet ensuite de connaître le statut de la réponse et de lire les données renvoyées.
```

# Corrigé de l’exercice 2 — Préparer les données d’un formulaire

```html
<form id="contactForm">
  <div>
    <label for="nom">Nom</label>
    <input type="text" id="nom" name="nom" />
  </div>

  <div>
    <label for="email">Adresse courriel</label>
    <input type="email" id="email" name="email" />
  </div>

  <div>
    <label for="message">Message</label>
    <textarea id="message" name="message"></textarea>
  </div>

  <button type="submit">Envoyer</button>
</form>

<script>
  const form = document.querySelector("#contactForm");

  form.addEventListener("submit", function (event) {
    event.preventDefault();

    const formData = new FormData(form);
    const donnees = Object.fromEntries(formData.entries());

    console.log(donnees);
  });
</script>
```

## Explication

`FormData` récupère les données du formulaire à partir des attributs `name`.

Par exemple :

```html
<input type="text" name="nom" />
```

permet d’obtenir une donnée appelée `nom`.

```html
<form id="contactForm">
  <div>

  </div>

  <div>

    <input type="email" id="email" name="email" />
  </div>

  <div>
    <label for="message">Message</label>
    <textarea id="message" name="message"></textarea>
  </div>

  <button type="submit">Envoyer</button>
</form>

<script>
  const form = document.querySelector("#contactForm");

  form.addEventListener("submit", function (event) {
    event.preventDefault();

    const formData = new FormData(form);
    const donnees = Object.fromEntries(formData.entries());

    const donneesJSON = JSON.stringify(donnees);

    console.log("Objet JavaScript :", donnees);
    console.log("Chaîne JSON :", donneesJSON);
  });
</script>
```

## Réponses aux questions

Un objet JavaScript est une structure utilisée directement dans le code JavaScript.

Exemple :

```javascript
{
  nom: "Sara",
  email: "sara@email.com"
}
```

Une chaîne JSON est une version textuelle de cet objet, souvent utilisée pour l’envoyer à une API.

Exemple :

```json
{ "nom": "Sara", "email": "sara@email.com" }
```

On transforme souvent les données en JSON avant de les envoyer à une API parce que les requêtes HTTP transmettent des données sous forme de texte structuré.

# Corrigé de l’exercice 5 — Utiliser `try...catch`

```html
<form id="contactForm">
  <label for="nom">Nom</label>
  <input type="text" id="nom" name="nom" />

  <label for="message">Message</label>
  <textarea id="message" name="message"></textarea>

  <button type="submit">Envoyer</button>
</form>

<p id="resultat"></p>

<script>
  const form = document.querySelector("#contactForm");
  const resultat = document.querySelector("#resultat");

  form.addEventListener("submit", async function (event) {
    event.preventDefault();

    const formData = new FormData(form);
    const donnees = Object.fromEntries(formData.entries());

    resultat.textContent = "Tentative d’envoi du formulaire...";

    try {
      const response = await fetch("/api/contact", {
        method: "POST",
        headers: {
          "Content-Type": "application/json",
        },
        body: JSON.stringify(donnees),
      });

      console.log(response);
    } catch (error) {
      resultat.textContent = "Impossible de contacter le serveur.";
      console.error(error);
    }
  });
</script>
```

## Réponse à la question

`try...catch` est important parce qu’une requête réseau peut échouer. Par exemple, le serveur peut être fermé, l’URL peut être incorrecte ou la connexion peut être interrompue.

# Corrigé de l’exercice 9 — Créer une route API en GET

## `server.js`

```javascript
import express from "express";

const app = express();
const PORT = 3000;

app.use(express.static("public"));

app.get("/api/status", function (req, res) {
  res.json({
    message: "API fonctionnelle.",
  });
});

app.listen(PORT, function () {
  console.log(`Serveur lancé sur http://localhost:${PORT}`);
});
```

## Réponse à la question

`res.send()` peut envoyer du texte, du HTML ou d’autres types de réponses simples.

Exemple :

```javascript
res.send("Bonjour");
```

`res.json()` envoie une réponse au format JSON.

Exemple :

```javascript
res.json({
  message: "Bonjour",
});
```

Pour une API, on utilise souvent `res.json()`.

# Corrigé de l’exercice 13 — Créer une route POST avec Express

## `server.js`

```javascript
import express from "express";

const app = express();
const PORT = 3000;

app.use(express.static("public"));
app.use(express.json());

app.post("/api/contact", function (req, res) {
  res.json({
    message: "Route POST reçue.",
  });
});

app.listen(PORT, function () {
  console.log(`Serveur lancé sur http://localhost:${PORT}`);
});
```

## Réponse à la question

L’URL dans `fetch` doit correspondre exactement à la route Express. Si le client envoie vers :

```javascript
fetch("/api/contact");
```

le serveur doit avoir :

```javascript
app.post("/api/contact", ...)
```

Sinon, le serveur ne saura pas quelle route exécuter.

## Réponses aux questions

# Corrigé de l’exercice 16 — Vérifier `response.ok`

## `public/script.js`

```javascript
const form = document.querySelector("#contactForm");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    if (!response.ok) {
      resultat.textContent = data.message || "Une erreur est survenue.";
      return;
    }

    resultat.textContent = data.message || "Message envoyé avec succès.";
  } catch (error) {
    resultat.textContent = "Impossible de contacter le serveur.";
    console.error(error);
  }
});
```

## Réponse à la question

`try...catch` ne suffit pas toujours pour gérer les erreurs HTTP, parce que `fetch` peut recevoir une réponse avec un statut `400` ou `500` sans déclencher automatiquement le `catch`.

C’est pour cela qu’on vérifie :

```javascript
response.ok;
```

# Corrigé de l’exercice 17 — Validation simple côté serveur

## `server.js`

```javascript
app.post("/api/contact", function (req, res) {
  const nom = req.body.nom?.trim();
  const email = req.body.email?.trim();
  const message = req.body.message?.trim();

  if (!nom || !email || !message) {
    return res.status(400).json({
      message: "Tous les champs sont obligatoires.",
    });
  }

  res.status(200).json({
    message: "Votre message a bien été reçu.",
  });
});
```

## Réponses aux questions

Il faut valider côté serveur parce que la validation côté client peut être contournée.

`res.status(400)` indique que la requête envoyée par le client est incorrecte ou incomplète.

Le `return` permet d’arrêter la fonction immédiatement après l’envoi de la réponse d’erreur. Cela évite que le serveur continue et tente d’envoyer une autre réponse.

# Corrigé de l’exercice 18 — Afficher les erreurs serveur côté client

## `public/script.js`

```javascript
const form = document.querySelector("#contactForm");
const resultat = document.querySelector("#resultat");

form.addEventListener("submit", async function (event) {
  event.preventDefault();

  resultat.textContent = "";

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  try {
    const response = await fetch("/api/contact", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
      },
      body: JSON.stringify(donnees),
    });

    const data = await response.json();

    if (!response.ok) {
      resultat.textContent = data.message;
      return;
    }

    resultat.textContent = data.message;
    form.reset();
  } catch (error) {
    resultat.textContent = "Impossible de contacter le serveur.";
    console.error(error);
  }
});
```

## Résultat attendu

Si le formulaire est incomplet :

```txt
Tous les champs sont obligatoires.
```

Si le formulaire est complet :

```txt
Votre message a bien été reçu.
```

# Corrigé de l’exercice 19 — Comparer validation client et validation serveur

```javascript
/*
1. La validation côté client sert à aider l’utilisateur immédiatement.
   Elle permet d’afficher des messages rapidement sans attendre la réponse du serveur.

2. La validation côté serveur sert à protéger l’application et à vérifier les données réellement reçues.

3. La validation côté client ne suffit pas, car elle peut être contournée.
   Un utilisateur peut désactiver JavaScript, modifier le HTML ou envoyer une requête autrement.

4. La validation côté serveur est obligatoire dans une vraie application, car le serveur ne doit jamais faire confiance aveuglément aux données reçues.

5. Non, ce n’est pas une bonne idée d’envoyer un mot de passe dans la console avec console.log().
   Les informations sensibles ne doivent pas être affichées dans les logs.
*/
```

# Corrigé de l’exercice 22 — Comprendre `express.urlencoded()`

## HTML

```html
<form action="/contact-classique" method="POST">
  <label for="nomClassique">Nom</label>
  <input type="text" id="nomClassique" name="nom" />

  <label for="messageClassique">Message</label>
  <textarea id="messageClassique" name="message"></textarea>

  <button type="submit">Envoyer classiquement</button>
</form>
```

## Express

```javascript
app.use(express.urlencoded({ extended: true }));

app.post("/contact-classique", function (req, res) {
  console.log(req.body);

  res.send("Formulaire classique reçu.");
});
```

## Réponses aux questions

On utilise `express.urlencoded()` parce qu’un formulaire HTML classique n’envoie pas les données en JSON. Il les envoie dans un format encodé que ce middleware permet de lire.

`express.json()` sert à lire des données JSON.

Ce formulaire n’utilise pas `fetch`. Il est soumis directement par le navigateur grâce à `action` et `method`.

# Corrigé de l’exercice 23 — Identifier les erreurs fréquentes

## Code client problématique

```javascript
const response = await fetch("/api/contact", {
  method: "POST",
  body: JSON.stringify(donnees),
});
```

## Code serveur problématique

```javascript
app.post("/contact", function (req, res) {
  console.log(req.body);
});
```

## Problèmes possibles

Premier problème : les routes ne correspondent pas.

Le client envoie vers :

```txt
/api/contact
```

mais le serveur écoute :

```txt
/contact
```

Correction possible :

```javascript
app.post("/api/contact", ...)
```

Deuxième problème : le client n’envoie pas le header `Content-Type`.

Correction :

```javascript
headers: {
  "Content-Type": "application/json"
}
```

Troisième problème : le serveur doit utiliser `express.json()` pour lire le JSON.

Correction :

```javascript
app.use(express.json());
```

Quatrième problème : le serveur ne répond pas au client.

Correction :

```javascript
res.json({
  message: "Données reçues.",
});
```

`response.json()` ne fonctionne pas ici parce que le serveur ne renvoie pas une réponse JSON.

```javascript
response.text();
```

Le client et le serveur doivent être d’accord sur le format. Si le serveur renvoie du texte, le client doit lire du texte. Si le serveur renvoie du JSON, le client peut utiliser `response.json()`.

# Corrigé de l’exercice 30 — Synthèse écrite

## Réponses

1. `fetch` sert à envoyer des requêtes HTTP depuis le navigateur vers un serveur.

2. On utilise souvent `POST` avec un formulaire parce que le formulaire envoie des données au serveur.

3. Le header `"Content-Type": "application/json"` indique que les données envoyées sont au format JSON.

4. `JSON.stringify()` transforme un objet JavaScript en chaîne JSON.

5. `express.json()` permet à Express de lire les données JSON reçues dans le corps de la requête.

6. Côté serveur, les données reçues se trouvent dans `req.body`.

7. `res.json()` permet au serveur de répondre au client avec une réponse JSON.

8. On doit valider côté serveur parce que les validations côté client peuvent être contournées.

9. `express.json()` lit les données JSON. `express.urlencoded()` lit les données envoyées par un formulaire HTML classique.

10. Une erreur du type `Cannot POST /api/contact` signifie souvent qu’aucune route POST ne correspond à cette adresse côté serveur.

# Ce qu’il faut retenir

La structure côté client est généralement :

```javascript
form.addEventListener("submit", async function (event) {
  event.preventDefault();

  const formData = new FormData(form);
  const donnees = Object.fromEntries(formData.entries());

  const response = await fetch("/api/contact", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
    },
    body: JSON.stringify(donnees),
  });

  const data = await response.json();
});
```

La structure côté serveur est généralement :

```javascript
app.use(express.json());

app.post("/api/contact", function (req, res) {
  console.log(req.body);

  res.json({
    message: "Données reçues.",
  });
});
```

La logique complète est :

```txt
Le formulaire est soumis.
JavaScript intercepte la soumission.
Les données sont récupérées avec FormData.
Les données deviennent un objet JavaScript.
L’objet est transformé en JSON.
fetch envoie les données au serveur.
Express reçoit la requête.
express.json() lit le body.
Les données sont disponibles dans req.body.
Le serveur valide les données.
Le serveur répond avec res.json().
Le navigateur affiche la réponse.
```
